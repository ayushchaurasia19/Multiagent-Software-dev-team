# AgentState Documentation

The `AgentState` is the core data structure that gets passed around between every node in your LangGraph workflow. You can think of it as the shared "memory" of your multi-agent system. 

Here is a detailed breakdown of each attribute defined in `src/state.py`:

> [!NOTE]
> When a node in LangGraph finishes its execution, it returns a dictionary containing *only the keys it wants to update*. LangGraph then merges these updates into the global `AgentState` before passing it to the next node.

### 1. `requirements` (Type: `str`)
*   **What it is:** The initial prompt or goal provided by the user.
*   **Who uses it:** 
    *   **Planner Agent:** Reads it to break the goal down into smaller tasks.
    *   **Backend & Frontend Agents:** Read it to understand the overall context of what they need to build.

### 2. `backend_tasks` & `frontend_tasks` (Type: `List[str]`)
*   **What it is:** Structured lists of sub-tasks generated by the Planner, strictly separated by domain.
*   **Who uses it:** 
    *   **Planner Agent:** Creates and populates these lists.
    *   **Backend & Frontend Agents:** Read their respective list to know exactly what files and features to implement.
    *   **Reviewer Agent:** Cross-references the written code against both task lists to ensure everything was completed.

### 3. `review_status` (Type: `str`)
*   **What it is:** A flag indicating whether the codebase passes inspection. Can be either `"approved"` or `"revise"`.
*   **Who uses it:** 
    *   **Reviewer Agent:** Sets this flag based on its evaluation.
    *   **LangGraph Edge Router:** Uses this flag to decide whether to dispatch revisions to the parallel agents or finish the workflow.

### 4. `review_feedback` (Type: `List[str]`)
*   **What it is:** A structured list of specific problems found during code review. It is empty if the status is `"approved"`.
*   **Who uses it:** 
    *   **Reviewer Agent:** Populates this array with criticisms, prefixing each with `[BACKEND]` or `[FRONTEND]`.
    *   **Backend & Frontend Agents:** Read this list during a revision cycle. They automatically skip execution if the feedback does not contain their specific prefix tag.

### 5. `review_count` (Type: `int`)
*   **What it is:** A counter that tracks how many revision cycles have occurred.
*   **Who uses it:** 
    *   **Reviewer Agent:** Increments this by 1 every time it evaluates the code.
    *   **LangGraph Edge Router:** Checks this counter to prevent infinite loops (enforcing `MAX_REVIEWS`).

### 6. `messages` (Type: `Annotated[List[str], operator.add]`)
*   **What it is:** A simple array of strings used exclusively for terminal logging and high-level debugging.
*   **Who uses it:** All agents append plain English status updates to this list.
> [!IMPORTANT]
> Because the Backend and Frontend agents execute in parallel, this attribute uses a specific LangGraph reducer (`operator.add`) so that concurrent updates are safely merged without crashing.

### 7. `backend_messages`, `frontend_messages` & `tester_messages` (Type: `Annotated[Sequence[BaseMessage], add_messages]`)
*   **What it is:** The actual LangChain message history (contains `SystemMessage`, `HumanMessage`, `AIMessage`, and `ToolMessage` objects), isolated by agent domain.
*   **Who uses it:** 
    *   **Agents & Tool Executors:** Each agent (Backend, Frontend, Tester) only sees and appends to its specific message array. This strict isolation prevents the LLMs from getting confused by each other's tool calls and context.

### 8. `backend_done`, `frontend_done` & `tester_done` (Type: `bool`)
*   **What it is:** Synchronization flags used to pause workflow execution until parallel branches or sequential testing steps complete.
*   **Who uses it:** 
    *   **Marker Nodes:** Set to `True` when their respective agent finishes execution.
    *   **Join Node / Router:** Checks if all required flags are `True` (e.g., waiting for Tester to finish after Backend). If so, it proceeds to the Reviewer.
    *   **Reviewer Agent:** Resets these flags to `False` if a revision is required.

### 9. `test_report` (Type: `str`)
*   **What it is:** A text summary of the automated tests written and executed, including any bugs or missing coverage identified.
*   **Who uses it:**
    *   **Tester Agent:** Generates this report based on its evaluation of the codebase.
    *   **Reviewer Agent:** Reads this report to factor automated testing results into its final evaluation and revision feedback.
